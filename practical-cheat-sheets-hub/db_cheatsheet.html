Keys & Integrity
Primary Key · Foreign Key
Primary Key (PK)
Unique ID per record. Never null. One per table. e.g. StudentID
Foreign Key (FK)
Column in Table B that references PK of Table A. Links tables. e.g. StudentID in Orders
Referential Integrity
Rule: FK must always point to an existing PK — or be NULL.
  • Prevents orphan records — orders with no customer
  • Cascade Delete: Deleting parent deletes children
  • Cascade Update: PK change flows to FK
  • Enforced automatically once set in DBMS
Normalization
Normalization — Overview
Goal: Eliminate data redundancy & avoid update / insert / delete anomalies.
1
1NF — Atomic Values
No repeating groups. Each cell = one value. Table has a PK.
2
2NF — Full Dependency
Must be 1NF. Every non-key field depends on the whole PK (remove partial dependency).
3
3NF — No Transitive
Must be 2NF. Non-key fields must depend only on the PK, not on each other.

Indexing
Index = Like a book's index — helps DBMS find rows fast without scanning every record.
✔ Benefits
Faster SELECT queries
Faster ORDER BY sorting
✘ Trade-offs
Slows INSERT/UPDATE/DELETE
Uses extra disk space
PK is always indexed automatically.

Interface & Analysis Tools
Forms
Purpose: Data INPUT — user-friendly interface to enter/edit records
  • Controls: textbox, dropdown, checkbox, button
  • Can have subforms (related records inside)
  • Validates data before saving to table
  • Hides raw table from end-user
Reports
Purpose: Formatted OUTPUT — print-ready, read-only presentation of data
  • Can group, sort, and total data
  • Has Header / Footer / Page sections
  • No editing — output only (print / PDF)
  • Uses query or table as data source
Pivot Table / Chart
Purpose: Summarise & analyse large datasets by rotating rows & columns.
  • When to use: Comparing categories across time
  • Pivot Chart = Visual (bar, line, pie) version
  • In Access: PivotTable View on queries / tables
